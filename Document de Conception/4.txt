## 12. Tests automatisés — Vitest

### 12.1 Pourquoi des tests automatisés

VOH a 101 000+ lignes de code. Chaque modification peut casser autre chose sans qu'on le remarque. Tester manuellement à chaque changement c'est impossible. Les tests automatisés garantissent que ce qui marche continue de marcher.

### 12.2 Outil choisi : Vitest

Vitest est le framework de test standard pour les projets JavaScript modernes. Natif ES modules, ultra-rapide, et peut tester dans un vrai navigateur via Playwright.

### 12.3 5 niveaux de tests

| Niveau | Type | Ce qu'il teste | Vitesse |
|--------|------|---------------|---------|
| 1 | Unitaire | Chaque module isolé | Rapide |
| 2 | Intégration | Modules ensemble (créer → copier → undo) | Moyen |
| 3 | Canvas | Rendu Pixi.js et interactions (navigateur réel) | Lent |
| 4 | API | Toutes les méthodes publiques | Moyen |
| 5 | E2E | Scénarios complets + stress | Lent |

### 12.4 Couverture minimale ciblée

| Module | Couverture minimale |
|--------|-------------------|
| Core managers (History, Selection, Clipboard, Pages) | 90%+ |
| BaseObject + TypeRegistry | 90%+ |
| RuleEngine | 85%+ |
| Save/Load (ProjectManager) | 95%+ |
| FileManager (import + cache) | 90%+ |
| CodeGenerator | 80%+ |
| RenderEngine (Pixi.js) | 60%+ |
| InteractionManager | 70%+ |
| Editor UI | 40%+ |
| Total projet | 75%+ |

### 12.5 Quand les tests sont écrits

Les tests sont écrits en même temps que le code, à chaque phase. Pas de "phase tests" séparée.

## 13. Zoom & scroll — Deux niveaux

### 13.1 Zoom de la zone (viewport)

La zone est le viewport. Elle possède son propre système de zoom et de scroll qui affecte **tous les canvas** qu'elle contient. C'est un zoom de navigation : on zoome/dézoome pour voir l'ensemble des canvas dans la zone, et on scroll pour se déplacer.

Ce zoom est déjà implémenté dans Core-Zone.js via CSS (zoom + overflow scroll).

### 13.2 Zoom interne d'un canvas (contenu de la page)

Chaque canvas a ses 5 layers Pixi.js (fond, image, grille, objets, overlay). Un ViewTransform appliqué sur le Container parent de ces 5 layers permet de zoomer/scroller **à l'intérieur** d'un canvas pour naviguer dans ses objets.

```javascript
class ViewTransform {
  constructor(pixiStage) {
    this._stage = pixiStage;  // Le Container Pixi.js parent des 5 layers
    this.zoom = 1.0;          // 1.0 = 100%
    this.scrollX = 0;
    this.scrollY = 0;
    this.minZoom = 0.1;       // 10%
    this.maxZoom = 10.0;      // 1000%
  }

  // Applique le zoom et scroll sur le Container Pixi.js
  apply() {
    this._stage.scale.set(this.zoom);
    this._stage.position.set(-this.scrollX * this.zoom, -this.scrollY * this.zoom);
  }

  worldToScreen(worldX, worldY) {
    return {
      x: (worldX - this.scrollX) * this.zoom,
      y: (worldY - this.scrollY) * this.zoom
    };
  }

  screenToWorld(screenX, screenY) {
    return {
      x: screenX / this.zoom + this.scrollX,
      y: screenY / this.zoom + this.scrollY
    };
  }
}
```

Le zoom et le scroll internes sont appliqués une seule fois sur le Container parent — tous les enfants (les 5 layers et leurs contenus) sont transformés automatiquement par le GPU.

### 13.3 Contrôles utilisateur (à définir)

- Molette : zoom centré sur le curseur (dans le canvas actif)
- Clic milieu + drag : déplacement panoramique (pan)
- Ctrl + 0 : reset zoom à 100%
- Ctrl + +/- : zoom par paliers

## 15. Risques et points d'attention

### 15.1 Performance du moteur de règles

Risque : évaluation des règles trop lente sur mousemove avec 500+ objets.
Mitigation : pré-compilation en fonctions JS natives + indexation par événement. Si malgré tout c'est plus lent, option de "mode direct" qui court-circuite le moteur.

### 15.2 Complétude de la reconstruction

Risque : oublier des fonctionnalités de la v4 pendant la réécriture.
Mitigation : la v4 sert de cahier des charges vivant. Chaque fonctionnalité de la v4 est cochée dans une checklist. Tests de non-régression : charger un projet v4 converti, vérifier que tout fonctionne identiquement.

### 15.3 Compatibilité des projets existants

Risque : les fichiers .voh de la v4 ne chargent plus en v5.
Mitigation : la v5 détecte la version du fichier et propose la migration avec dialogue de confirmation.

### 15.4 Taille du chantier

Risque : projet trop ambitieux qui n'aboutit pas.
Mitigation : les phases sont indépendantes. La Phase 1 (Zone) est déjà fonctionnelle. On peut s'arrêter après n'importe quelle phase.

### 15.5 Wrapper desktop — Choix différé

Risque : le choix du wrapper desktop est reporté. Quand on y arrivera (Phase 8), le paysage des frameworks aura peut-être changé, ou le framework choisi pourrait avoir ses propres problèmes (comme Neutralinojs en v4).
Mitigation : l'architecture Core/Editor est 100% indépendante du wrapper. Le même code tourne dans un navigateur et dans n'importe quel wrapper. Si un framework pose problème, on peut en changer sans toucher au Core ni à l'Editor. Le risque est isolé dans la couche wrapper uniquement.

### 15.6 Open source — Concurrence et fork

Risque : le Core est open source. Un concurrent pourrait forker le projet et créer un produit concurrent.
Mitigation : l'avance technique et la connaissance du projet sont les vrais avantages compétitifs. Un fork sans la vision conceptuelle et l'expertise accumulée ne vaut pas grand-chose. De plus, l'utilisation de bibliothèques tierces (Pixi.js sous licence MIT) impose de respecter les licences — être open source est cohérent avec cet écosystème. La communauté open source apporte aussi des contributions, des retours de bugs, et de la visibilité.

### 15.7 Auto-save et perte de données

Risque : l'auto-save écrase une version correcte avec une version contenant des erreurs de l'utilisateur.
Mitigation : l'auto-save est une copie du .voh dans un dossier séparé (Auto-Save/), pas un écrasement du .voh principal. La rotation de slots (défaut : 10) permet de remonter dans le temps. Le .voh principal n'est modifié que par Ctrl+S.

### 15.8 Dépendance à Pixi.js

Risque : Pixi.js est une bibliothèque tierce. Si Playco arrête le développement ou introduit des changements cassants, VOH est impacté.
Mitigation : Pixi.js est sous licence MIT — on peut forker si nécessaire. Le projet est très actif (release régulières, large communauté, utilisé en production par des entreprises). VOH utilise Pixi.js comme moteur de rendu interne uniquement — l'API publique VOH ne dépend pas de Pixi.js. Si un remplacement est nécessaire un jour, seuls les fichiers `engine/Render*.js` sont impactés. De plus, la version de Pixi.js est fixée dans le `package.json` — pas de mise à jour automatique non voulue.
